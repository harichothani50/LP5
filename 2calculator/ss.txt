# 1. FIRST UNDERSTAND THE MAIN PROBLEM

Before CORBA existed:

Different programming languages could NOT communicate easily.

Example:

| Language | Problem                   |
| -------- | ------------------------- |
| Java     | Different runtime         |
| C++      | Different memory handling |
| Python   | Different architecture    |

Suppose:

* Client written in Java
* Server written in C++

How will they communicate?

That was the problem.

---

# 2. WHAT IS CORBA?

CORBA =
Common Object Request Broker Architecture

Created by:
OMG organization
(Object Management Group)

---

# MAIN GOAL OF CORBA

Allow programs written in:

* Java
* C++
* Python
* Ada
* COBOL

to communicate remotely.

---

# REAL LIFE EXAMPLE

Imagine:

Different countries:

* English speakers
* Japanese speakers
* Hindi speakers

Need communication.

Solution:
translator/interpreter.

CORBA acts as universal translator.

---

# 3. MOST IMPORTANT CONCEPT — ORB

ORB =
Object Request Broker

ORB is the HEART of CORBA.

---

# WHAT DOES ORB DO?

ORB handles:

* communication
* object locating
* request forwarding
* data conversion
* network transmission

Client never directly talks to server.

Everything passes through ORB.

---

# REAL LIFE EXAMPLE

Customer → Receptionist → Manager

Customer doesn't directly access manager.

Receptionist handles communication.

ORB = Receptionist.

---

# 4. CORBA ARCHITECTURE

Your project architecture:

```text
CLIENT
   |
   v
ORB
   |
Name Service
   |
SERVER OBJECT
```

---

# 5. WHAT IS IDL?

IDL =
Interface Definition Language

MOST IMPORTANT CONCEPT.

---

# WHY IDL EXISTS?

Different languages have different syntax.

Example:

Java:

```java id="7c8u5q"
int add(int a, int b)
```

C++:

```cpp id="gkpqdu"
int add(int a, int b);
```

Need common format.

IDL provides universal interface description.

# FILE 1 — `Calculator.idl`



```idl id="9z6tt1"
module CalculatorApp {
    interface Calculator {
        float add(in float a, in float b);
        float sub(in float a, in float b);
        float mul(in float a, in float b);
        float div(in float a, in float b);
    };
};
```

---

# FIRST UNDERSTAND IDL

IDL is NOT Java.

IDL is universal definition language.

It ONLY defines:

* methods
* parameters
* return types

NOT implementation.

---

# LINE 1

```idl id="8j7lwp"
module CalculatorApp
```

---

# WHAT IS MODULE?

Module is like:

* package
* namespace
* container

Used to organize interfaces.

---

# WHY MODULE IS IMPORTANT?

Suppose:
two applications both have:

```idl id="f50qzt"
Calculator
```

Without modules:
name conflict happens.

Module prevents conflicts.

---

# LINE 2

```idl id="5y4d1v"
interface Calculator
```

---

# SAME AS JAVA INTERFACE

Defines methods only.

NO logic.

---

# LINE 3

```idl id="4xz6f5"
float add(in float a, in float b);
```

---

# BREAKING THIS COMPLETELY

---

# `float`

Return type.

Returns decimal value.

---

# `add`

Method name.

---

# `in`

VERY IMPORTANT.

CORBA supports:

* in
* out
* inout

---

# WHAT IS `in`?

Means:
value travels:
CLIENT → SERVER

ONLY input.

---

# EXAMPLE

```idl id="39kvj8"
add(10,5)
```

10 and 5 go from client to server.

---

# WHY CORBA HAS `in/out/inout`?

Because CORBA supports many languages.

Some languages handle parameters differently.

Main POA manager.

# WHAT IS `resolve_initial_references()`?

ORB provides initial services.

Like:

* RootPOA
* NameService

This function retrieves them.

---

# WHAT IS `narrow()`?

MOST IMPORTANT CORBA CONCEPT.

---

# WHY NARROW EXISTS?

CORBA initially returns:
generic object.

Need conversion into proper type.

`narrow()` safely converts it.

---

# SIMILAR TO CASTING

Java casting:

```java id="xldv3t"
(MyClass)obj
```

CORBA uses:

```java id="yuwjlwm"
Helper.narrow(obj)
```

because remote objects are special.

---

# ACTIVATE POA MANAGER

```java id="u5kb6m"
rootpoa.the_POAManager().activate();
```

---

# WHY ACTIVATION NEEDED?

Without activation:
POA will NOT process requests.

It is like:
turning ON the server.

---

# CREATE SERVANT

```java id="0r8j5u"
CalculatorImpl calcImpl = new CalculatorImpl();
```

Creates actual implementation object.

---

# CONVERT SERVANT TO CORBA OBJECT

```java id="rwjlwm"
org.omg.CORBA.Object ref = rootpoa.servant_to_reference(calcImpl);
```

---

# WHY THIS IS NEEDED?

Servant itself cannot directly travel over network.

Need CORBA object reference.

This creates network-accessible reference.

---

# NARROW AGAIN

```java id="6v87x2"

```

Converts generic CORBA reference into Calculator type.

# DIFFERENCE BETWEEN RMI AND CORBA

| Feature              | RMI            | CORBA                          |
| -------------------- | -------------- | ------------------------------ |
| Language dependent   | Yes            | No                             |
| Uses Java only       | Yes            | No                             |
| Uses IDL             | No             | Yes                            |
| Complexity           | Simple         | Complex                        |
| Performance          | Faster in Java | Flexible                       |
| Platform independent | Partial        | Strong                         |
| Used in              | Java systems   | Enterprise distributed systems |
